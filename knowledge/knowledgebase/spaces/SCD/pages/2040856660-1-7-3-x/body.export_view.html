Follow this post for any live updates published after the 1.7.3 Release on February 8, 2016, and before the next numbered release.

Feb 9, 10:45 AM CST - Return ticket load fix

Fixed an issue with child items in a bundle. When loading a return ticket that has a taxable child item, the tax calculation would fail to run, causing the return item list to not load.

Feb 10, 12:45 PM CST - Summary Report failing to load

Fixed an issue where the Summary Report would occasionally fail to load, due to a bug in the way empty "Tax Details" were handled.

Feb 10, 2:30 PM CST - Return item "tax details" were not saving properly

The "tax details" value (e.g. "Sales Tax: $2.43 Other Tax: $0.56") was not being saved properly for return items as of the 1.7.3 release on February 9 (1 day ago). This would cause the "returned tax detail" figure(s) on the tax report to be incorrect, and would cause return tickets with more than one return to sometimes fail to load properly. The bug has been fixed, and any return items with invalid tax detail values have had the proper values restored.

Feb 16, 3:28 PM CST - Ticket Export, Billing Agent values

Ticket export changed so that the "Billing Agent" column represents the Billing Agent associated with the ticket, rather than the Billing Agent associated as the customer's default.

Feb 16, 3:57 PM CST - Ticket Attribute Changelog

Restored ticket attribute change logging that result from ticket saves from the ticket edit screen.

Feb 18, 1:04 PM CST - NEW FEATURE Inventory Details - New Page Filters

We added 2 new filters to the Inventory Details page. A date filter, where you can filter by added date, updated date, or status updated date, and an "Inactive items only" filter where you can filter for the inventory records that belong to inactive catalog items.

Feb 18, 4:44 PM CST - Ticket item backorder linking to wrong PO fixed

If you have an item on a PO in the ordered status, and at the same time sell the same item on a ticket while out of stock, rather than setting the item added to the ticket as "backordered" it links the item to the purchase order and advances the item to the "ordered" status. This is meant to help prevent special ordering a backordered item because you already have it on order. However, in some rare cases, if you had the item ordered on an old purchase order that was already received or rejected in full, but not advanced to the "reconciled" status, it would link the new backordered item from the ticket to the old purchase order which cannot fulfill the item. This should no longer occur as of this fix.

Feb 19, 12:37 AM CST - NEW FEATURE - Save inventory item origin PO link

The inventory item purchase order link can change when an item is involved in one or more internal purchase order transfers, and we wanted to add an attribute that stores the original purchase order the inventory item was first added on. This origin PO value remains unchanged so that you always retain a link to the original purchase order the inventory item was received on. The origin PO is displayed on the inventory item inspect, view, details, and details export. The inventory item origin PO will only be tracked for new items going forward as of this update.

Feb 23, 12:55 PM CST - Return Repair replacement serialized item update

When you return and replace a serialized item on a Return Repair ticket, it asks you to enter the serial number for the item before finishing the ticket. If you had serialized items at different price points, it wouldn't always accept the serial number entered. This intermittent problem has been fixed, and it will now accept all valid serial numbers entered for the replacement item.

Feb 29, 2:30 PM CST - Return Repair by Tech Report resolution filter fix

Using Ticket Resolution filter would prevent the report from loading. This has been fixed.

Feb 29, 2:53 PM CST - Ignore restock fee for zero dollar items

Restock fees would not work properly if you processed a subsequent return for a return repair item (i.e. returning a return repair). The fix involved ignoring the restock fee for zero dollar returns.

Feb 29, 3:02 PM CST - Inventory Reports supplier filter add "no supplier" option

For tradeins it is possible to have inventory items with no supplier value set. You can now filter for these items on your inventory reports using the "No supplier set" option in the supplier filter. In addition, if you select "All suppliers" will now properly include all items including those inventory items where no supplier was set.

Mar 2, 12:25 PM CST - Permission structure updates

With this update, location based permissions are enforced more strictly. We also resolved an issue with the bulk edit permission on the inventory details page where only admin users reliably had the permission to do bulk edits. Lastly, shift leads and inventory managers will now be able to click on the "Reports" link and view the reports they have access to, where previously they could not navigate to the Reports section directly. To accommodate non-managers navigating to the reports section, we also created a reports section "landing page," rather than redirecting to the Revenue Report by default.

Mar 4, 5:40 PM CST - Discount percentage permission fix

Staff percentage discounts were not enforcing the max discount level properly, enforcing percentage as a dollar amount instead of a percentage. This issue has been resolved. Note: you may need to reload the ticket edit screen or reset your browser file cache for the update to apply.

Mar 10, 11:43 AM CST - Suppress sending notifications to customers for refurb tickets

If you had a ticket automated notification setup to send to the {customer} template tag, you could get notifications sent out to the customer for refurbish tickets. Since refurbish tickets are for "internal" repairs, it makes sense to suppress any notifications setup for delivery to the customer. As of this update, no more notifications will be queued up for delivery to the {customer} for refurb tickets.

Mar 10, 12:02 AM CST - Fix tax report excel export

The export was choking whenever records without any tax detail values were present. This issue has been fixed and exporting the tax report should work normally now.

Mar 10, 1:44 PM CST - Fix issue with cash payments less than 1.00

Cash payments entered for less than 1.00 using the "Add Transaction Form" (not including the "Quick Cash" input) were not being allowed. The maximum payment allowable was getting not set properly, so any payment less than 1.00 using that form was being rejected, saying "payment must be less than or equal to 0.00." This behavior has been fixed, and processing cash payments for less than 1.00 should now work properly. NOTE: clicking the browser reload/refresh page button on the ticket edit screen may be required to receive the fix. If that doesn't work, empty your browser file cache.

Mar 15, 10:38 AM CST - Serial number unique validation for single item edit

If you edited an existing inventory item, you could change the serial number to match the serial number for another instock item, which should not be permissible.The individual inventory item edit action was the only place where this was possible, and we have now closed the loophole.

Mar 28, 10:59 AM CST - Automated Notification recipients

Exclude non-active staff members from notifications. Also, notifications configured for admin & accountant roles should work more reliably.

Apr 13, 1:04 PM CST - Prevent Merging a catalog item into a deleted catalog item

The Catalog Item Suggest Merge feature now excludes catalog items flagged as inactive or deleted. This feature is triggered from the purchase order edit screen or the catalog item edit screen when a supplier sku is entered that is already associated with a different catalog item, and when triggered asks "This supplier sku is associated with [Item XYZ], would you like to merge this item with[Item XYZ]?" 

Apr 18, 11:38 AM CST - Resolve Credit Card swipe issues on Google Chrome version 50.0.2661.75

We received many reports of people being unable to get past the "Invalid credit card entry" entry when processing credit cards in RepairQ using Google Chrome. This error is produced when RepairQ does not get a valid input from the credit card reader. The issue was related to the latest Google Chrome update, where they changed the way the browser process hidden inputs. We just published an update that resolves the issue for the latest version of Chrome. NOTE: you may need to reload/refresh the page while on the ticket edit screen to pick up this update in your browser.
