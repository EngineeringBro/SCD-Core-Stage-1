Validate Your DNS Records for Email Authentication

After adding the required DNS records, please allow up to 24 hours for the changes to fully take effect. If you encounter any issues with record validation, you can view detailed error messages in Mandrill for more information. Additionally, you may use third-party resources to verify your records or obtain further insights if needed:

  • MXToolbox (mxtoolbox.com)
    MXToolbox offers tools for checking DNS, MX, SPF, and other records to help identify issues quickly.

  • DNS Checker (dnschecker.org)
    DNS Checker allows you to verify DNS record propagation across multiple global locations, which is useful to ensure records are correctly updated worldwide.

  • Google Admin Toolbox Dig (toolbox.googleapps.com)
    This Google tool provides a straightforward interface to look up DNS records and confirm they are set up correctly.

  • SPF Validator (http://www.kitterman.com/spf/validate.html)
    Use the SPF Validator to check if your SPF records are properly configured. Enter your domain name in the first text box and click Get SPF Record to view a diagnostic of your SPF setup. The validator will search for a TXT record with the required SPF information. If your domain has an "SPF" type record, consider adding a matching "TXT" record for broader compatibility.

    • If you already have an SPF record, edit the existing one instead of adding a new one, as the SPF specification requires only one TXT record containing SPF information for each domain.

  • DKIMCore Validator (http://dkimcore.org/tools/keycheck.html)
    To check the validity of your DKIM record, use the DKIMCore Validator. Enter “mandrill” as the selector and your domain name in the designated fields. This tool will confirm that your DKIM record is correctly configured for secure email delivery.

DKIM Record Formatting Issues

If you've added the DKIM record but still receive an error indicating that it’s missing, your DNS provider may require a different formatting. The DKIM record provided by Mandrill includes semicolons that are escaped with a backslash. For example, the record will start like this:

 v=DKIM1\; k=rsa\; p=...

and ends like this:

 \;

  • v=DKIM1\; specifies the version.

  • k=rsa\; indicates the key type, which is usually RSA.

  • p= followed by a long alphanumeric string is the public key itself, which can be hundreds of characters long and typically ends with the final character of that key.

If you've added the DKIM record but it’s still showing as missing or invalid, here are a few things to check:

  • Backslash and Semicolon Formatting
    Some DNS providers don’t require semicolons to be escaped with a backslash. If you’re experiencing issues, try removing the backslashes directly before the semicolons at the beginning and end of the DKIM record. This adjustment can help make the record compatible with your provider's formatting requirements.

  • Publishing and Propagation Times
    Keep in mind that DNS providers vary in how quickly they publish and push updates. New records often validate within 10-15 minutes, while changes to existing records can take longer, depending on the provider and the TTL (Time To Live) set for the record. If validation issues persist, allow additional time for the record to propagate and verify again later.
